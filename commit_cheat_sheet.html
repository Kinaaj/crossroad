Commit Cheat-Sheet

The template for writing a semantic commit is (note that seperator for these parts is the blank line!):

<type>(<scope>): <subject>

<body>

<footer>

And here is the explanation for each part:

<type> (Mandatory)

The <type> is one keyword and it communicates the main intention of the change.

Keywords Description SemVer
feat A new feature for the user Minor
fix A bug fix for the user Patch
docs Documentation changes Patch
style Formatting, missing semi-colons, etc; no code change Patch
refactor Refactoring production code (no new features/fixes) Patch
perf Code change that improves performance Patch
test Adding missing tests, refactoring tests None
chore Updating build tasks, package manager configs, etc None
ci Changes to CI configuration files and scripts None
revert Reverting a previous commit Patch

<scope>

A <scope> should be one word and may be provided to a commitâ€™s type, to provide additional contextual information and is contained within parenthesis.

feat(parser): add ability to parse arrays
fix(api): handle empty response body

<subject> (Mandatory)

The <subject> contains a succinct description of the change. Adhere to these rules:

  • Use the imperative, present tense: "change" not "changed" nor "changes".
  • Don't capitalize the first letter.
  • No dot (.) at the end.
  • Limit to 50 characters.

<body>

Paragraph of detailed explanation. Use the imperative, present tense. The body should include the motivation for the change and contrast this with previous behavior. It should contain max 72 characters per line.

<footer>

The <footer> is optional. It is used for meta-information about the commit, primarily for tracking issues or indicating major breaking changes.

1. Breaking Changes

A commit that has a footer starting with BREAKING CHANGE:, or appends a ! after the type/scope, introduces a breaking API change (correlating with Major in Semantic Versioning).

feat(auth)!: remove support for Basic Auth

BREAKING CHANGE: Basic Auth is no longer supported, please use Bearer tokens.

2. Referencing Issues

If the commit closes a specific issue, listing it here will automatically close it in GitHub, GitLab, or Jira.

Closes #123
Fixes #405

To commit using this full format in the terminal:

git commit -m "feat: add new login" -m "This adds the ability..."
